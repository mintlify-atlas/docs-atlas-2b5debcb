---
title: PKG Configuration
description: Configuration options for creating macOS PKG installers
---

The top-level `pkg` key contains options for building [PKG](https://en.wikipedia.org/wiki/.pkg) (macOS installer component package) installers.

## Basic Example

```json
{
  "pkg": {
    "installLocation": "/Applications",
    "allowAnywhere": true,
    "allowCurrentUserHome": true,
    "allowRootDirectory": true,
    "license": "license.txt"
  }
}
```

## Configuration Options

### Installation Settings

<ParamField path="installLocation" type="string" default="/Applications">
  The install location. [Do not use it](https://stackoverflow.com/questions/12863944/how-do-you-specify-a-default-install-location-to-home-with-pkgbuild) to create per-user package.
  
  `/Applications` would install it as expected into `/Applications` if the local system domain is chosen, or into `$HOME/Applications` if the home installation is chosen.
  
  <Note>
    Mostly never you will need to change this option.
  </Note>
</ParamField>

<ParamField path="allowAnywhere" type="boolean" default="true">
  Whether can be installed at the root of any volume, including non-system volumes. Otherwise, it cannot be installed at the root of a volume.
  
  Corresponds to [enable_anywhere](https://developer.apple.com/library/content/documentation/DeveloperTools/Reference/DistributionDefinitionRef/Chapters/Distribution_XML_Ref.html#//apple_ref/doc/uid/TP40005370-CH100-SW70).
</ParamField>

<ParamField path="allowCurrentUserHome" type="boolean" default="true">
  Whether can be installed into the current user's home directory.
  
  A home directory installation is done as the current user (not as root), and it cannot write outside of the home directory. If the product cannot be installed in the user's home directory and be not completely functional from user's home directory.
  
  Corresponds to [enable_currentUserHome](https://developer.apple.com/library/content/documentation/DeveloperTools/Reference/DistributionDefinitionRef/Chapters/Distribution_XML_Ref.html#//apple_ref/doc/uid/TP40005370-CH100-SW70).
</ParamField>

<ParamField path="allowRootDirectory" type="boolean" default="true">
  Whether can be installed into the root directory. Should usually be `true` unless the product can be installed only to the user's home directory.
  
  Corresponds to [enable_localSystem](https://developer.apple.com/library/content/documentation/DeveloperTools/Reference/DistributionDefinitionRef/Chapters/Distribution_XML_Ref.html#//apple_ref/doc/uid/TP40005370-CH100-SW70).
</ParamField>

### Installer Behavior

<ParamField path="isRelocatable" type="boolean" default="true">
  Install bundle over previous version if moved by user?
</ParamField>

<ParamField path="isVersionChecked" type="boolean" default="true">
  Don't install bundle if newer version on disk?
</ParamField>

<ParamField path="hasStrictIdentifier" type="boolean" default="true">
  Require identical bundle identifiers at install path?
</ParamField>

<ParamField path="overwriteAction" type="'upgrade' | 'update'" default="upgrade">
  Specifies how an existing version of the bundle on disk should be handled when the version in the package is installed.
  
  **`upgrade`:** The bundle in the package atomically replaces any version on disk; this has the effect of deleting old paths that no longer exist in the new version of the bundle.
  
  **`update`:** The bundle in the package overwrites the version on disk, and any files not contained in the package will be left intact; this is appropriate when you are delivering an update-only package.
  
  <Note>
    Another effect of `update` is that the package bundle will not be installed at all if there is not already a version on disk; this allows a package to deliver an update for an app that the user might have deleted.
  </Note>
</ParamField>

### Scripts

<ParamField path="scripts" type="string" default="build/pkg-scripts">
  The scripts directory, relative to `build` (build resources directory).
  
  The scripts can be in any language so long as the files are marked executable and have the appropriate shebang indicating the path to the interpreter.
  
  <Warning>
    Scripts are required to be executable (`chmod +x file`).
  </Warning>
  
  See [Scripting in installer packages](http://macinstallers.blogspot.de/2012/07/scripting-in-installer-packages.html).
</ParamField>

### Signing and License

<ParamField path="identity" type="string">
  The name of certificate to use when signing. Consider using environment variables [CSC_LINK or CSC_NAME](/code-signing) instead of specifying this option.
</ParamField>

<ParamField path="license" type="string">
  The path to EULA license file. Defaults to `license.txt` or `eula.txt` (or uppercase variants).
  
  In addition to `txt`, `rtf` and `html` supported (don't forget to use `target="_blank"` for links).
</ParamField>

### Customization

<ParamField path="background" type="object">
  Options for the background image for the installer.
  
  **Properties:**
  
  <ParamField path="background.file" type="string">
    Path to the image to use as an installer background.
  </ParamField>
  
  <ParamField path="background.alignment" type="string" default="center">
    Alignment of the background image.
    
    Options are: `center`, `left`, `right`, `top`, `bottom`, `topleft`, `topright`, `bottomleft`, `bottomright`
  </ParamField>
  
  <ParamField path="background.scaling" type="string" default="tofit">
    Scaling of the background image.
    
    Options are: `tofit`, `none`, `proportional`
  </ParamField>
</ParamField>

<ParamField path="welcome" type="string">
  The path to the welcome file. This may be used to customize the text on the Introduction page of the installer.
</ParamField>

<ParamField path="conclusion" type="string">
  The path to the conclusion file. This may be used to customize the text on the final "Summary" page of the installer.
</ParamField>

<ParamField path="mustClose" type="string[]">
  Identifies applications that must be closed before the package is installed.
  
  Corresponds to [must-close](https://developer.apple.com/library/archive/documentation/DeveloperTools/Reference/DistributionDefinitionRef/Chapters/Distribution_XML_Ref.html#//apple_ref/doc/uid/TP40005370-CH100-SW77).
  
  **Example:**
  
  ```json
  {
    "mustClose": [
      "com.example.myapp"
    ]
  }
  ```
</ParamField>

### Advanced Options

<ParamField path="extraPkgsDir" type="string">
  The extra component packages directory (relative to build resources directory) for MacOS product archive.
  
  Autoscans directory for any `.pkg` files and adds to `productbuild` command as `--package-path` and `--package` accordingly.
</ParamField>

## Example with Custom Scripts

Create a `build/pkg-scripts` directory with the following scripts:

### preinstall

```bash
#!/bin/bash

echo "Running preinstall script"
# Close running instances
pkill -x "MyApp" || true

exit 0
```

### postinstall

```bash
#!/bin/bash

echo "Running postinstall script"
# Set permissions or perform cleanup
chmod -R 755 "/Applications/MyApp.app"

exit 0
```

Make them executable:

```bash
chmod +x build/pkg-scripts/preinstall
chmod +x build/pkg-scripts/postinstall
```
